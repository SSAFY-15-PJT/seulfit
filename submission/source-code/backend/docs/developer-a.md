# 개발자 A 담당 범위

## 역할
개발자 A는 SeulPick의 인프라 및 알고리즘 영역을 담당한다.

개발자 A의 핵심 책임은 위치, 카드, 혜택, 소비 데이터를 조합해 카드 추천 결과를 만드는 것이다.

## 담당 영역
- Kakao Local API 연동.
- OpenWeatherMap API 연동.
- 하이퍼로컬 상권 요약.
- 반경 500m 인프라 분석.
- 지도 요약 API.
- 날씨 큐레이션 API.
- 카드 데이터 크롤링 파이프라인.
- 카드 마스터 데이터 정규화.
- Graph DB 추천 데이터 모델 및 Neo4j 연동.
- Seul-Score 알고리즘.
- 월 단위 카드 혜택 시뮬레이션.
- 대시보드에서 바로 사용할 수 있는 추천 API 응답 구조.

## 담당하지 않는 영역
- 영수증 또는 카드 소비 리포트 이미지의 멀티모달 파싱.
- OpenAI/Gemini 이미지 API 호출.
- 로그인, 회원가입, JWT 또는 세션 인증.
- 커뮤니티 게시글 CRUD.
- LLM 기반 게시글 검열.
- 프로필 UI 전체 구현. 단, 추천에 필요한 보유 카드나 선호 위치 필드는 예외적으로 다룰 수 있다.

## 개발자 B와의 공유 계약
개발자 B는 이미지에서 추출한 월간 소비 데이터를 아래 JSON 형태로 전달한다.

```json
{
  "cafe": 120000,
  "convenience": 45000,
  "mart": 320000,
  "dining": 130000,
  "delivery": 50000,
  "shopping": 90000,
  "etc": 50000
}
```

가맹점 제한 혜택을 계산하려면 선택적으로 거래 상세도 전달한다.

```json
{
  "transactions": [
    {
      "category": "cafe",
      "merchant_name": "스타벅스 강남역점",
      "amount": 6500,
      "transaction_date": "2026-06-21",
      "transaction_time": "18:30",
      "channel": "offline"
    }
  ]
}
```

`merchant_name`이 없으면 특정 가맹점 전용 혜택은 집계 소비액으로 추정하지
않고 0원으로 처리한다.

`transaction_date`는 `YYYY-MM-DD`, `transaction_time`은 `HH:MM`,
`channel`은 `online` 또는 `offline`을 사용한다. 거래일자가 없으면 일
한도·일 횟수 혜택을, 거래시각이 없으면 시간대 혜택을, 채널이 없으면
온라인·오프라인 전용 혜택을 0원 처리한다.

개발자 A는 이 데이터를 아래 API에서 소비한다.

```text
POST /api/v1/hyperlocal/simulate/
```

## 현재 결정 사항
- Graph DB는 실제로 연동한다. 단, 백엔드 경계와 데이터 계약을 먼저 고정한 뒤 진행한다.
- 대상 Graph DB는 Neo4j로 한다.
- 카드 데이터는 정적 mock이 아니라 실제 카드 데이터를 크롤링해서 사용한다.
- 소비 카테고리는 크롤링한 카드 혜택 카테고리를 검토한 뒤 확장할 수 있다.
- MVP 지도 분석 카테고리는 아래 네 가지로 고정한다.
  - `cafe`
  - `convenience`
  - `mart`
  - `dining`
  - `delivery`

기존 클라이언트의 `food` 입력은 하위 호환을 위해 `dining`으로 처리한다.
신규 클라이언트는 외식과 배달을 각각 `dining`, `delivery`로 전달한다.
- Seul-Score에는 아래 조건을 포함한다.
  - 약관 기반 월 예상 총혜택.
  - 월 환산 연회비를 차감한 예상 순혜택.
  - 실제 혜택 금액과 분리된 주변 가맹점 기반 지역 적합도.
  - 전월 실적 조건.
  - 카테고리별 한도와 월 통합 할인 한도.
  - 사용자 소비 데이터가 없을 때의 콜드스타트 소비 프로필.
  - 보유 카드 표시 상태.

## 추천 비즈니스 목표
SeulPick 추천 시스템은 카드사 마진이나 카드 발급 전환율이 아니라 사용자의
설명 가능한 실질 절약액을 우선한다.

```text
사용자의 소비 패턴과 생활권에서 사용할 수 있는 카드 중,
전월 실적과 할인 한도, 연회비를 반영한 월 예상 순혜택이 가장 큰 카드를 추천한다.
```

정렬 우선순위는 아래와 같다.

1. 전월 실적 충족 여부
2. 월 예상 순혜택
3. 지역 적합도
4. 월 예상 총혜택

## 보유 카드 표시 계약
추천된 카드가 사용자가 이미 보유한 카드라면 응답에 아래 값을 포함한다.

```json
{
  "is_owned": true,
  "badge": "보유중인 카드"
}
```

프론트엔드 카드 추천 UI는 카드 이미지 또는 카드 이름 근처에 `보유중인 카드` 문구를 표시한다.

## 구현 경계
개발자 A는 추천 계산에 필요한 필드와 API를 추가할 수 있다. 다만 전체 프로필 관리 기능을 구현하지는 않는다.

MVP에서는 보유 카드 ID를 시뮬레이션 요청에 직접 포함하거나, 이후 간단한 백엔드 관계 테이블에서 불러오는 방식으로 시작할 수 있다.
