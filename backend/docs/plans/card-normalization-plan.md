# 카드 데이터 정규화 작업 계획

## 진행 상태

- 상태: 신한카드 16개 확보, 5개 active, 11개 검토 대기
- 구현 시작: 2026-06-20
- 구현 완료: 미완료
- 검증 결과: 신한카드 5개 실제 SQLite 추천 후보 및 한도 계산 확인

## 확정 사항

### 기준 저장소

```text
Django ORM + SQLite
```

SQLite는 카드 상품, 혜택 원문, 정규화 결과, 크롤링 상태와 이미지 경로의
기준 저장소로 사용한다. Neo4j는 검증된 카드와 혜택 관계를 동기화한 추천 후보
탐색용 저장소로 사용한다.

### 하루 수집 범위

수집 채널은 다음 6개로 한다.

- 신한카드
- KB국민카드
- 삼성카드
- 현대카드
- 카카오뱅크
- 토스뱅크

하루 작업의 목표는 다음과 같다.

```text
기본정보, 이미지, 혜택 원문 수집: 40~80개
추천 계산 가능 상태로 정규화: 15~30개
나머지 복잡한 상품: review_required
```

신한카드, KB국민카드, 삼성카드와 현대카드는 신용·체크카드를 합쳐 채널별
최대 15개를 수집한다. 카카오뱅크와 토스뱅크는 공식 페이지에서 확인 가능한
자체 체크카드와 제휴 신용카드 상품을 수집한다.

카드 이미지의 디자인 종류는 별도 카드 상품으로 세지 않는다. 동일한 혜택과
약관을 가진 디자인 변형은 하나의 상품과 여러 이미지 변형으로 저장한다.

### 예상 소요 시간

실제 40~80개 페이지와 이미지 다운로드 실행은 일반적으로 수십 분 이내를
목표로 한다. 하루가 필요한 작업은 6개 사이트 구조 분석, 수집기 구현,
SQLite 적재, 이미지 검증, 중복 제거, 실패 처리와 결과 검수다.

### 비용 원칙

초기 크롤러는 Python 표준 HTTP 요청, Django ORM, SQLite, 로컬 파일 저장과
Neo4j Community Edition만 사용한다. 유료 크롤링 API, 유료 프록시, CAPTCHA
우회 서비스와 LLM API를 사용하지 않는다.

사이트가 자동 수집을 차단하면 유료 우회 수단을 도입하지 않고 해당 채널을
일시 중단한다. 배포 환경에서는 서버 트래픽과 이미지 디스크 사용량에 따른
인프라 비용이 발생할 수 있다.

### 자동 수집 정책

각 채널은 목록 또는 상품 페이지를 요청하기 전에 `robots.txt`를 확인한다.
일반 크롤러 접근이 허용되지 않으면 해당 작업을 `paused`로 기록하고 상품
페이지, 상세 페이지와 이미지를 수집하지 않는다.

카카오뱅크는 2026-06-20 확인 시 일반 사용자 에이전트에 `/` 전체를
`Disallow`하고 있어 자동 수집 대상에서 일시 제외한다.

신한카드는 `/pconts/html/card/` 경로를 명시적으로 허용한다. 첫 실제
수집 채널로 선정하고 공식 메인 페이지에 노출된 상품만 제한 수집한다.

## 중단 복구 계획

카드 한 건을 처리하기 직전에 `fetching`, 성공 직후 `success` 상태를 SQLite에
저장한다. 인터넷 단절이나 프로세스 종료가 발생해도 이미 완료한 항목은 유지한다.

상태:

```text
pending
fetching
success
retry_pending
paused
failed
```

복구 규칙:

- 연결 오류와 타임아웃은 최대 3회 재시도한다.
- 재시도 간격은 2초, 5초, 15초로 증가한다.
- 403 또는 429 응답은 해당 채널만 `paused` 처리한다.
- 성공한 URL은 같은 작업에서 다시 생성하거나 수집하지 않는다.
- 비정상 종료로 `fetching`에 남은 항목은 재개 시 `retry_pending`으로 복구한다.
- 이미지는 `.part` 임시 파일에 저장한 뒤 검증이 끝나야 최종 경로로 이동한다.
- 실패한 이미지의 임시 파일은 삭제한다.

개발 세션 또는 Codex 토큰이 종료되는 경우에도 런타임 진행 상태는 SQLite에
남는다. 계획 진행 상태, 마지막 검증 결과와 다음 작업은 이 문서의 작업 기록에
남겨 다음 세션에서 코드와 DB 상태를 함께 확인할 수 있게 한다.

예정 재개 명령:

```text
python manage.py crawl_cards --issuer shinhan --resume
python manage.py crawl_cards --issuer shinhan --resume --retry-failed
python manage.py crawl_cards --list-sources
```

## 작업 목표

크롤러, 추천 계산 코어와 Neo4j가 공통으로 사용할 카드 상품 및 혜택 데이터
계약을 확정하고 검증 계층을 구현한다.

```text
카드사 웹페이지
  -> 크롤링 원문
  -> 정규화 및 검증
  -> 승인된 카드 데이터
  -> Neo4j 적재
  -> 추천 계산 코어
```

## 작업 순서

1. 현재 추천 계산 코어가 요구하는 입력 필드를 점검한다.
2. `CardProduct`와 `BenefitRule` 정규화 구조를 확정한다.
3. 금액, 할인율, 횟수, 상태값과 `null` 처리 규칙을 정의한다.
4. 현재 계산 가능한 조건과 검토가 필요한 조건을 분리한다.
5. 크롤링 원문, 출처 URL과 수집 시각 보존 규칙을 정의한다.
6. 카드 이미지 다운로드, 검증과 중복 방지 규칙을 정의한다.
7. 실제 발급사와 수집 채널이 다른 제휴 카드의 식별 규칙을 정의한다.
8. 순수 Python 기반 스키마 검증 모듈을 추가한다.
9. 정상, 오류와 검토 대상 데이터에 대한 테스트를 추가한다.
10. 추천 계산 코어로 유입 가능한 데이터의 경계를 검증한다.
11. SQLite에서 검증된 데이터를 Neo4j로 동기화하는 계약을 정의한다.
12. 전체 `finance` 테스트와 Django 시스템 검사를 실행한다.
